1. How to Set Up the Project

1.1 Prerequisites

  • Node.js v18+
  • npm v9+
  • Docker Desktop (for local PostgreSQL)
  • Git

1.2 Step-by-Step Setup

Step 1: Fork & Clone the Repository

git clone https://github.com/<your_username>/saas-starter-kit.git
cd saas-starter-kit

Step 2: Install Dependencies

npm install

This installs all project dependencies including Next.js, Prisma, NextAuth.js, Stripe SDK, SAML Jackson, and more.

Step 3: Configure Environment Variables

cp .env.example .env

Then edit the .env file. Key variables you must configure:

VariableDescriptionDefault
DATABASE_URLPostgreSQL connection stringpostgresql://postgres:postgres@localhost:5432/saas-starter-kit
NEXTAUTH_SECRETEncryption secret for NextAuth.js(generate one)
NEXTAUTH_URLApplication URLhttp://localhost:4002
APP_URLSame as NEXTAUTH_URLhttp://localhost:4002
SMTP_HOST, SMTP_PORT, etc.Email configuration for magic links & invitations—
Note: Some features (Stripe, Svix, SAML SSO) require additional API keys. The app will work for core functionality without them, but those features will be disabled.

Step 4: Start PostgreSQL (Docker)

docker-compose up -d

This starts a PostgreSQL 16.4 container on port 5432 with the following defaults:

  • Database: saas-starter-kit
  • User: postgres
  • Password: postgres

Step 5: Push the Database Schema

npx prisma db push

This synchronizes the Prisma schema from prisma/schema.prisma to your PostgreSQL database, creating all tables, indexes, and enums.

What does this do? See Chapter 3 for a full explanation of "push schema."

Step 6: (Optional) Seed the Database

npx prisma db seed

Populates the database with sample data: 10 users, 5 teams, team memberships, and invitations.

Step 7: Start the Development Server

npm run dev

The app will be available at http://localhost:4002.

Step 8: (Optional) Open Prisma Studio

npx prisma studio

Opens a visual database browser at http://localhost:5555.

3. What Is "Push Schema" and Why Is It Needed?

3.1 Definition

"Push schema" refers to the Prisma CLI command:

npx prisma db push

This command synchronizes your Prisma schema (prisma/schema.prisma) directly to your database — creating, altering, or deleting tables, columns, indexes, and enums to match the schema definition.

3.2 How It Works

  1. Reads the schema.prisma file (the source of truth for your database structure)
  2. Connects to the database using DATABASE_URL from .env
  3. Compares the schema against the current database state
  4. Generates and executes SQL statements to make the database match the schema
  5. Reports any warnings (e.g., data loss if a column is removed)

3.3 Why Is It Needed?

1. First-Time Database Setup

After cloning the project and starting PostgreSQL, the database is empty. Running prisma db push creates all the tables, columns, indexes, and enums defined in the schema so the application can function.

Without this step: The app would crash with "relation does not exist" errors when trying to query users, teams, etc.

2. Schema Changes During Development

When you modify schema.prisma (add a field, change a type, add a model), prisma db push applies those changes immediately without creating migration files. This is ideal for rapid prototyping.

3. Keeping Prisma Client Updated

After pushing schema changes, you also need to regenerate the Prisma Client:

npx prisma generate

This updates the TypeScript client library with type-safe methods for the new/changed models.

3.4 Push Schema vs. Migrations

Aspectprisma db pushprisma migrate
PurposeQuick sync for developmentVersion-controlled schema changes
Migration filesDoes NOT create anyCreates SQL migration files
Version historyNo historyFull change history
RollbackNot supportedSupported via migration files
Data loss warningsInteractive promptsChecked during generation
Production useNot recommendedRecommended (migrate deploy)
SpeedFast (no file generation)Slower (generates files)
Team collaborationHarder (no file to commit)Easier (files in version control)
In this project: The package.json build script runs prisma db push automatically: prisma generate && prisma db push && next build. This ensures the database schema is always up-to-date before building.

3.5 When to Use Each

  • Use prisma db push: During initial development, prototyping, or when you're the only developer working on the database schema.
  • Use prisma migrate dev: When you want to create a formal migration file to commit to version control, especially in a team setting.
  • Use prisma migrate deploy: In production/staging environments to apply migrations safely.

3.7 Summary

Push schema (prisma db push) is needed because:

  1. The database starts empty — tables must be created for the app to work.
  2. It provides a fast, iterative workflow during development.
  3. It keeps the database in sync with the Prisma schema as you make changes.
  4. It's integrated into the build pipeline (see package.json: "build": "prisma generate && prisma db push && next build").
Pro Tip: For team projects, consider switching to prisma migrate dev to generate migration files that can be reviewed in pull requests and applied consistently across environments.
